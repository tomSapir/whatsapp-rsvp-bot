# M8 — Streamlit app (in simple words)

## What this stage is

Everything so far is invisible machinery. M8 is the **host's control panel** —
the screen where you (the host) set up the event, manage the guest list, press
the big buttons, and watch the answers roll in. Built with Streamlit, which
turns a Python script into a web page: every widget call (`st.button`,
`st.metric`) draws something, and the script re-runs top to bottom on every
click.

## The most important decision: keep the UI dumb

Streamlit code is genuinely hard to test — it only behaves inside a running
Streamlit server. So we made the UI file (`host/dashboard.py`) as close to
**pure rendering** as possible, and pushed every decision into two ordinary
Python modules that tests can hammer:

- `app/actions.py` — what the buttons *do* (writes and sends).
- `app/reporting.py` — what the numbers *are* (read-only queries).

The UI file just calls them and draws the results. 117 tests run with zero
Streamlit involved.

## The three tabs

### 💍 Event setup

A form for the couple's names (English + Hebrew), the date, and an optional
image (saved into `data/uploads/`, with its *path* stored in the database —
databases hold pointers to big files, not the files). Saving "upserts": the
first save inserts the single event row, later saves update it — the
database's one-row rule from M1 is never violated.

### 👥 Guests

Add/edit/delete guests. Two guards fire **at the entry door**:

- The phone is run through M2's normalizer immediately. Garbage is rejected
  with a friendly message and *never stored*.
- The duplicate check compares **canonical** numbers — so `050-234-5678` and
  `+972 50-234-5678` are correctly caught as the same guest, even though the
  strings look different. This is the payoff of canonicalizing at every door.

Each guest row also shows context-dependent buttons (only where they make
sense — that's the UI's only "logic"):

- **Nudge** — only for "said Yes but never said how many". Sends a
  count-prompt template and re-arms the chat to expect details. This is the
  manual fix for the deliberate gap in M7 (the auto-job never chases these).
- **Re-invite** — only for declined/draft guests. Wipes the old RSVP, resets
  them to invited/awaiting-yes-no with a fresh reminder budget, and re-sends
  the invite. Re-inviting a *confirmed* guest is refused — it would throw
  away a real answer.

### 📊 Dashboard

- **Four buckets**: Coming / Declined / Awaiting reply / Not invited — these
  are exactly the four `status` values, counted.
- **Headcount, the honest way**: two numbers side by side — "known heads"
  (the sum of sizes people actually gave) and "coming, size unknown" (how
  many said Yes without a number). We never quietly pretend an unknown is
  0 or 1 — that's how headcounts silently lie.
- **Dietary needs** table, **activity feed** (M6's notifications, newest
  first), and two bulk buttons: *Send invites to all drafts* and *Remind
  non-responders now*.
- **CSV export** is the one place an unknown size becomes a number (a
  spreadsheet cell can't hold a shrug) — it exports as `1`, but with a
  `size_unknown` flag column set, so the substitution is visible.

## Two buttons, one difference

"Remind non-responders **now**" (the button) and the hourly M7 job send the
same template to the same kind of guest — but the button skips the
delay-window and max-count checks. Why? The job is a robot and needs rules;
the button is a deliberate human click, and the human is the boss. The button
still updates the same counters, though, so the robot's spacing stays correct
afterwards.

## Gotcha that bit us (again)

A made-up test phone number (`052-123-4567`) failed validation — the
`phonenumbers` library knows which ranges really exist in Israel. Third time
this lesson appears: **test data must be as real as the validator is strict.**
