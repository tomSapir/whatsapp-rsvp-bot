# M4 — Webhook ingestion (in simple words)

## What this stage is

The **webhook** is the front door of the bot. When a guest sends a WhatsApp
message, Meta's servers take it and POST it to *our* web address. M4 is the
code that answers that door: `app/webhook.py`, built with FastAPI.

The door is on the public internet, so most of this stage is about being
careful: check who is knocking, don't process the same knock twice, and never
keep Meta waiting.

## What we did, step by step

### 1. The handshake (`GET /webhook`)

Before Meta sends us anything, it checks the address really belongs to us.
It sends a GET request with two things: a secret word (`hub.verify_token`)
and a random challenge string (`hub.challenge`).

Our code: if the secret word matches the one we chose, echo the challenge
back. If not, answer 403 ("forbidden"). That's the whole handshake. One
small detail: Meta wants the challenge back as a bare string, not wrapped in
JSON quotes, so we return a plain-text response.

### 2. The signature gate (`POST /webhook`)

The handshake only guards the sign-up. After that, **anyone** on the internet
can POST to our address and pretend to be Meta. So every POST must prove
itself.

How the proof works: Meta and we share a secret (the **App Secret**). Meta
takes the exact bytes of the request body, mixes them with the secret using a
formula called **HMAC-SHA256**, and puts the result in a header
(`X-Hub-Signature-256`). We run the *same* formula on the body we received.
Same result → really Meta. Different → fake, drop it with 403.

Two traps we avoided:

- **Use the raw bytes.** We must run the formula on the body *exactly as it
  arrived*, before any JSON parsing. If we parsed and re-built the JSON,
  even one moved space would change the result.
- **Compare in constant time.** A naive `==` on strings stops at the first
  wrong letter, so a wrong-first-letter guess fails *slightly faster* than a
  wrong-last-letter guess. An attacker can measure that and crack the
  signature letter by letter. Python's `hmac.compare_digest` always takes the
  same time, so it leaks nothing.

This check runs **first** — before parsing, before touching the database.

### 3. Answer fast, work later

Meta gives us only a few seconds to answer "200 OK". If we're slow, it thinks
we're down and **sends the same event again**. Later (M5) we'll call OpenAI
while handling a message — far too slow to do before answering.

So we split it: the endpoint checks the signature, says "200, got it"
immediately, and hands the real work to a **background task** (FastAPI runs
it right after the response goes out).

### 4. Don't process the same message twice

Meta promises "at-least-once" delivery — meaning duplicates *will* happen.
Our defense uses the database rule from M1: the `messages` table refuses two
rows with the same WhatsApp message id (`wa_message_id` is UNIQUE).

So the very **first** thing we do with a message is try to insert its row.
- Insert worked → first time we see it → continue.
- Database said "already exists" → duplicate delivery → stop quietly.

Order matters here: because the insert comes *before* any other action (like
notifying the host), a duplicate can never cause a double notification.

### 5. Two kinds of events on one door

Meta sends two different things to the same address:
- `messages[]` — a guest actually wrote to us → process.
- `statuses[]` — receipts about *our own* sends ("delivered", "read") →
  just log and ignore. If we treated these as replies, every message we send
  would "answer itself".

The code simply looks at which list exists in the payload and branches.

### 6. Who is the sender?

Meta gives the sender as `wa_id` — the phone number as bare digits, no `+`
(e.g. `972502345678`). We pass it through M2's `normalize_phone`, which turns
it into the same `+972…` form we store, then look up the invitation with that
phone.

- **Found** → save the message linked to that invitation. (M5 will pick it
  up from here and actually understand it.)
- **Not found** (or the digits don't even parse) → still save the message
  (with no invitation) and tell the host: "message from an unknown number".
  We **never** auto-create an invitation — only the host adds guests.

### 7. The seams (how this stays testable)

The router is built by a **factory function** that takes its dependencies as
arguments: the verify token, the app secret, the database session maker, and
a `notify` function. Production will pass the real ones (M9); tests pass a
temp database and a notify that just appends to a list. Also, `notify` is a
placeholder for now (it logs) — M6 will replace it with the dashboard
activity feed without touching this file's logic.

### 8. The tests

`tests/test_webhook.py` spins up the app with FastAPI's `TestClient` (a fake
browser — no real server) and checks every rule above:

- handshake: right token → challenge echoed; wrong token → 403
- correct signature → accepted; wrong/missing signature → 403 and the
  database stays empty
- the same event POSTed twice → stored once, host notified once
- a status receipt → 200, but nothing stored
- known sender → message linked to their invitation
- unknown sender → message stored unlinked + host notified

One bug we hit while writing them: the test used a made-up phone number, and
the `phonenumbers` library rejected it as not a real number — so the "known"
guest was never matched. Lesson: test phone numbers must be *really valid*
ranges, not just digits that look right.
