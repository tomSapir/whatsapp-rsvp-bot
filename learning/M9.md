# M9 — Integration & wiring (in simple words)

## What this stage is

Every earlier stage built a piece that *asks for* its dependencies instead of
creating them. M9 is where someone finally *hands them over* — once with real
parts (production), once with fake parts (the end-to-end tests). Plus the
optional "grade the real LLM" suite.

The two remaining steps in STEPS (creating the WhatsApp templates and the
live smoke test) are manual — they happen in Meta's website and on a phone,
not in code.

## Part 1 — the wiring (`app/main.py`)

A tiny new file with one job: **build the object graph**. It reads the
settings, opens the database, constructs the *real* Graph API client, the
*real* OpenAI parser, the feed notifier, plugs them all into the webhook app,
and attaches the hourly reminder scheduler to the server's start/stop events.

This pattern has a name — the **composition root**: one single place in the
whole program where concrete things are chosen and connected. Everything
else stays "I work with whatever you hand me." That's why tests could fake
everything so easily for eight milestones.

One Python detail: we run it as
`uvicorn app.main:create_app --factory` — the `--factory` flag means "call
this function to *get* the app" instead of importing a ready-made `app`
object. Why bother? Because building the app reads secrets and touches the
disk. Keeping that inside a function (not at import time) is what lets the
test suite import every module without a single secret set.

The missing link also got connected here: until now the webhook saved
incoming messages and stopped (M4 ended with a "M5 hooks in here" comment).
Now a matched message continues into the conversation engine — button taps
go the deterministic route, text goes through the parser.

## Part 2 — the end-to-end tests

All the earlier tests checked one layer at a time. The e2e tests check the
**whole pipe** in one breath, playing Meta:

1. Build the app exactly like `main.py` does — but hand it the fakes.
2. POST a byte-for-byte realistic webhook payload, properly HMAC-signed.
3. Assert on the *far end*: the guest's status flipped, the RSVP row is
   right, the follow-up question went out in the right language, the feed
   got its entry.

The scenarios: button Yes (Hebrew guest — follow-up arrives in Hebrew),
button No, Hebrew free text that completes the RSVP in one shot ("כן! נגיע
3, אחת צמחונית"), English free-text decline, a question (nothing changes,
host notified), a status receipt (ignored), an unknown number, and the
duplicate-delivery case — the same signed event POSTed twice ends with
exactly one RSVP, one follow-up, one feed entry.

Why these matter even with 117 unit tests already green: unit tests prove
each gear turns; the e2e test proves the gears actually mesh — that the
webhook *calls* the engine, in the same database session, with the right
arguments.

## Part 3 — the opt-in LLM eval

Every normal test uses the stub parser, because real LLM calls are slow,
cost money, and can answer differently twice. But *someone* has to check the
real model + our prompt actually understand Hebrew RSVPs. That's
`tests/test_llm_eval.py`: ten realistic phrases (5 Hebrew, 5 English),
sent to the real OpenAI API, checking the **intent** and the explicit
numbers come back right.

It's skipped unless you explicitly run:

```
RUN_LLM_EVAL=1 OPENAI_API_KEY=sk-... pytest tests/test_llm_eval.py
```

So the default suite stays free, fast, and deterministic — and the paid,
fuzzy check is one command away when you want it.

## What's left (manual, on Meta's side)

1. In **WhatsApp Manager**, create the bilingual templates the code refers
   to by name: `wedding_invite` and `rsvp_reminder` (and `rsvp_details_nudge`
   for the nudge button) — with Yes/No quick-reply buttons — then wait for
   approval.
2. The **smoke test**: run all three processes (uvicorn, streamlit, tunnel),
   register the webhook URL in Meta's dashboard, send yourself an invite,
   tap Yes on your own phone, and watch yourself appear on the dashboard.
