# M5 — Parsing & conversation state machine (in simple words)

## What this stage is

The **brain**. M4 catches a guest's message and saves it; M5 decides what the
message *means* and what to do about it. Two new files:

- `app/parser.py` — turns free text into structured data (using OpenAI).
- `app/conversation.py` — takes that data and updates the guest's RSVP,
  following strict rules.

## Part 1 — the parser

### The problem

A guest can write anything: "yes!! we're 3, one vegan", "מצטערים, לא נוכל",
"is there parking?". Code can't `if` its way through human language. This is
the one place we use an LLM.

### The trick: tool calling (function calling)

We don't ask the model to "answer" — free-form prose would be a pain to read
back. Instead we tell it: "here is a function called `record_rsvp_reply`,
with these exact fields — you **must** call it." The model then has no choice
but to answer in our shape:

```
intent, attending, party_size, dietary, note, confidence
```

### The rules baked into the design

- **`intent` is a closed list**: `rsvp_yes`, `rsvp_no`, `provide_details`,
  `change`, `question`, `other`. Six options, nothing else. The code branches
  on this one field — and code can only branch on values it knows in advance.
- **`null` means "the guest didn't say it".** The prompt orders the model:
  set a field only if the guest *explicitly* said it; never guess.
- **`confidence` is logged, never obeyed.** Models are bad judges of their
  own certainty, so we write it to the log and ignore it for decisions.
- **Trust nothing.** Before using the model's answer, we validate every
  field's type. Unknown intent? `party_size` is `"three"` instead of `3`?
  → `ParseError`. A funny detail: in Python, `True` *is* a kind of integer,
  so we explicitly refuse booleans as a party size.

### The same seam trick as M3

`ReplyParser` is the contract. `OpenAIReplyParser` is the real one.
`StubReplyParser` is the fake for tests: a dictionary saying "this text means
that result". So every state-machine test runs with zero network and zero
randomness — we test *our rules*, not the model's mood.

## Part 2 — the conversation engine

### Two kinds of replies

- **A button tap (Yes/No)** is deterministic — the guest literally pressed a
  button we made. No LLM needed; we map it straight to the rules. (Both
  English "Yes/No" and Hebrew "כן/לא" buttons are recognized.)
- **Free text** goes through the parser first, then the same rules.

### The state machine

Each invitation carries two separate fields (set together, in one
transaction):

- `status` — the *outcome*: invited → confirmed / declined.
- `conversation_state` — *where the chat stands*: waiting for yes/no,
  waiting for the head-count, or done.

The flow in plain words:

- **Yes** (button or clear text) → confirmed. Know the head-count already?
  Done. Don't know it? → state `awaiting_details`, and the bot **sends a
  follow-up question** ("How many of you? Any dietary needs?") in the guest's
  language. We only ask when *entering* that state — tapping Yes twice
  doesn't nag twice.
- **No** → declined, done. And the head-count is **wiped**: the database has
  a hard rule (from M1) that a decline can never carry a head-count, so a
  "would have been 4" never pollutes the totals.
- **Details** ("we're 4, one gluten-free") → fill in only the fields the
  guest gave. **`null` never overwrites**: if they mention only diet, the
  head-count we already know stays.
- **Change of mind** → the latest message always wins, no time limit.
  No→Yes reopens the head-count question; Yes→No clears it.
- **Question or anything unclear** → the bot touches **nothing** and tells
  the host: "Dana asked: is there parking?". A human answers humans; the bot
  only does paperwork it is sure about.

### Guard rails

- `party_size` must be 1–20. "We are 50" → not saved, host is told.
- A guest who sends details *before* ever saying yes or no ("3 of us"?) is
  ambiguous → touch nothing, tell the host.
- Parser blew up (`ParseError`) → touch nothing, tell the host.
- `attending` can only flip on an *explicit* yes/no — a vague message can
  never silently turn a Yes into a No.
- Every processed reply ends with one notification to the host, so the
  dashboard feed (M6) will show everything.

### Why the follow-up text message is allowed

WhatsApp only lets a business send free text within 24 hours of the guest's
last message. Here that's always true: we are *replying* to a message the
guest just sent, which itself reopened the 24-hour window. The follow-up is
also written to the message log, like every message in or out.

## The tests

`tests/test_conversation.py` has the heart of it: the **transition table as a
table of test cases** — each row is (situation before, incoming reply,
expected situation after). The plan's table from PLAN §5 became code almost
line for line, plus tests for every guard rail above.

`tests/test_parser.py` checks the validation (bad types rejected) and runs
`OpenAIReplyParser` against a tiny *fake* OpenAI client that returns canned
answers — proving we send the right request and survive a misbehaving model.
