# M3 — WhatsApp client (in simple words)

*(Written after the fact — this stage was done before the learning folder existed.)*

## What this stage is

The part that **sends** WhatsApp messages. The trick of this stage is not the
sending itself — it's building a "seam" so we can swap the real sender for a
fake one in tests.

## What we did

1. **A contract first.** `WhatsAppClient` is an *abstract class* — a promise
   that says: "anything calling itself a WhatsApp client must offer
   `send_template`, `send_text`, `send_interactive`." The rest of the app only
   ever talks to this promise, never to a concrete class. That is the seam.

2. **The real one: `GraphWhatsAppClient`.** Sending a WhatsApp message is just
   an HTTP POST to Meta's server (`graph.facebook.com/.../messages`) with our
   token in the header and the message as JSON. Meta answers with the new
   message's id — we keep it, because the webhook will later use the same id
   to spot duplicate deliveries.

3. **The fake one: `FakeWhatsAppClient`.** It sends nothing. It just writes
   every "send" into a list. A test can then look at the list and say: "yes,
   the app tried to send template X to +972…". No internet, instant, and the
   result is the same every run.

4. **Why bother with the fake?** Real sends cost a sandbox setup, are slow, and
   can fail for reasons that have nothing to do with our code. With the fake,
   we test *our* logic ("did the app decide to send the right thing?") and
   trust Meta to do the delivery.

The three "verbs" we send with, in WhatsApp's rules:
- **template** — a pre-approved message. The only thing allowed when *we*
  start the conversation (the invite, the reminder).
- **text** — free text. Allowed only within 24 hours after the guest's last
  message.
- **interactive** — messages with buttons or lists.
