# M6 — Notifications (in simple words)

## What this stage is

All through M4 and M5 the code kept saying "notify the host" — tell the human
that a guest replied, changed their mind, asked a question, wrote from an
unknown number, or sent a weird head-count. M6 is where those notifications
actually go: an **activity feed** that the dashboard (M8) will show, newest
first.

## Why not just WhatsApp the host?

Tempting, but no. To WhatsApp, the host is just another user — so the bot
hits the same wall it hits with guests: outside a 24-hour chat window it may
only send pre-approved *template* messages, and a template can't carry
free-form text like "Dana asked: is there parking?". So in v1 the feed lives
in the dashboard. (A free push channel like Telegram can be added later —
see "the seam" below for why that will be easy.)

## How it works

### A new table

`host_notifications` — the simplest table in the project: an id, the text of
the event, and a timestamp. Append-only, like the message log.

Why a database table and not, say, a list in memory? Because **two separate
programs** are involved: the FastAPI process *writes* events (when a webhook
arrives), and the Streamlit process *reads* them (to show the feed). They
can't share memory — but they already share the SQLite file. The WAL mode we
turned on back in M1 is exactly what makes this safe.

### The seam (the actual point of this stage)

Here's the nice part. In M4 and M5 we never hard-coded *how* to notify — the
webhook and the conversation engine just accept any function that takes a
string:

```python
notify("💬 Dana is coming — 3 people.")
```

So M6 only had to provide such a function. `FeedNotifier` is a small class
whose instances are *callable like a function* — calling one inserts a feed
row. It plugs straight into the `notify=` slot that M4 and M5 already have.
**Zero lines of existing code changed.**

That's the payoff of designing with seams: when we later want push
notifications too, we write another notifier class and swap it in at one
wiring point. The webhook and conversation code will never know.

One subtle choice: each notification commits in its **own** little database
transaction, immediately. We don't want a notification to be silently thrown
away (or half-written) because of whatever transaction the caller was in the
middle of.

### Reading the feed

`recent_notifications(session, limit)` returns the newest entries first —
this is what the dashboard will call. Small detail: we sort by `id`, not by
the timestamp, because the timestamp only stores whole seconds — two events
in the same second would have an unpredictable order, while ids always grow.

## The tests

For each of the five event types in the plan — a reply, an RSVP change, an
unknown number, a question, a validation failure — the test doesn't call the
notifier directly. It triggers the **real code path** that fires the event
(a button tap, a "we can't come" text, a webhook payload from a stranger…)
with `FeedNotifier` plugged in, then checks the feed got the right entry.
That way the tests prove the whole chain fits together, not just the last
link. Plus one test that the feed comes back newest-first and respects the
limit.
