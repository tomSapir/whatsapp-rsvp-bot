# M1 — Data layer (in simple words)

*(Written after the fact — this stage was done before the learning folder existed.)*

## What this stage is

The database: where the bot remembers everything. We use SQLite — the whole
database is just one file on disk, no server needed.

## What we did

1. **`app/db.py` — the connection.** This file knows how to open the database.
   Two important switches are flipped every time we connect:
   - **WAL mode** — lets two programs (the bot and the dashboard) use the same
     file at the same time without blocking each other.
   - **foreign_keys=ON** — SQLite, surprisingly, does *not* check links between
     tables by default. We turn the checking on.

2. **`app/models.py` — the four tables.** Each Python class becomes a table:
   - **Event** — the wedding itself (names, date, picture). Only one row is
     ever allowed — a database rule (`CHECK id = 1`) rejects a second one.
   - **Invitation** — one row per guest: name, phone, language, status
     (draft → invited → confirmed/declined), and where the chat stands.
   - **Rsvp** — the guest's answer: coming or not, how many people, diet notes.
   - **Message** — a diary of every WhatsApp message in or out. Never edited,
     only appended.

3. **Rules live in the database, not only in code.** Examples: a phone number
   can appear only once; each WhatsApp message id can appear only once; a guest
   who declined cannot also have a head-count. The database itself refuses bad
   rows. Why? Code can have bugs and forget to check — the database never
   forgets.

4. **Tests.** Each test creates a brand-new throwaway database in a temp
   folder, tries to break a rule (insert the same phone twice, etc.), and
   checks the database says **no**.
